# Style and conventions
- Language/toolchain: Go module `log-enricher`, targeting Go 1.24.x.
- Logging convention: prefer `slog` consistently; avoid mixing with the standard `log` package except for compatibility needs.
- API/architecture conventions from `AGENTS.md`:
  - Pass `context.Context` through network-bound operations.
  - Avoid goroutine leaks; every spawned goroutine needs a cancellation path.
  - Keep pool APIs type-safe around `*models.LogEntry`; avoid shadow model types.
  - Treat `main.go`, `internal/tailer/*`, `internal/processor/*`, `internal/pipeline/*`, `internal/backends/backend.go`, and `internal/models/models.go` as canonical unless deliberately migrated everywhere in one change.
  - When changing exported interfaces used across packages, update all call sites and tests in the same PR and verify no stale symbols remain.
- Testing conventions: prefer table-driven tests, especially for stage behavior and file-state edge cases.
- State/tailing safety: preserve rotation/truncation/file-identity behavior; any file identity logic changes should cover unchanged file, inode-changed rotation, truncation with same inode, and missing file.
- Configuration convention: behavior is primarily environment-variable driven. Pipeline stages are configured with ordered variables like `STAGE_<N>_TYPE`, optional `STAGE_<N>_APPLIES_TO`, and per-stage params `STAGE_<N>_<PARAM>`.
- Naming/organization pattern: small focused packages under `internal/`, exported constructors/functions in Go style, and backend/pipeline/tailer abstractions separated by package boundary.
